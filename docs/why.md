# Why documentation matters

Engineers and scientists already document constantly — just not always in a reusable way.

Documentation is what turns a one-off build into something that can be:
- **replicated**,
- **debugged**,
- **audited** (safety / traceability),
- and **improved** by the next person.

## Documentation is not “extra”

Good documentation reduces:
- rework,
- repeated questions,
- fragile tribal knowledge,
- and “archaeology” on old projects.

## Documentation starts on day 1

If you wait until the end:
- you’ll forget why decisions were made,
- you’ll lose intermediate photos,
- you’ll miss the “gotchas” that were obvious during assembly.

A practical rule:

> Start with a README and add to it *as you go*.

## The “happy medium”

Too little documentation = nobody can reproduce the work.  
Too much documentation too early = you may over-specify before you even know what users need.
